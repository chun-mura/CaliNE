# Security Review

CaliNE プロジェクトのコードに対してセキュリティレビューを実施するスキル。

## 使い方

```
/security-review
```

## レビュー対象ファイル

以下のファイルをすべて読み取り、セキュリティ上の問題を洗い出す:

- `src/` 配下の全 Python ファイル
- `scripts/` 配下の全 Python ファイル
- `.github/workflows/` 配下の全 YAML ファイル
- `requirements.txt`
- `.gitignore`

## レビュー観点

このプロジェクトは Microsoft Graph API、LINE Messaging API、Google Calendar API を使用し、GitHub Actions で日次実行される。以下の観点でレビューすること:

### 1. シークレット / 認証情報の漏洩

- 環境変数に格納されたトークン・シークレットがログ、標準出力、例外メッセージに露出していないか
- `.gitignore` にトークンファイル（`ms_token_cache.json`, `client_secret*.json`, `token.json`）が含まれているか
- GitHub Actions のログでシークレットがマスクされているか

### 2. GitHub Actions のセキュリティ

- `permissions` が最小権限（`contents: read`）に設定されているか
- シークレットが必要なステップのみに `env` として渡されているか（ジョブレベルではなくステップレベル）
- サードパーティ Action のバージョンがハッシュ固定またはメジャーバージョンタグで指定されているか

### 3. サプライチェーン

- `requirements.txt` で全パッケージのバージョンが固定されているか
- 既知の脆弱性を持つバージョンが使用されていないか

### 4. API 呼び出しのセキュリティ

- 外部 API 呼び出しに適切なタイムアウトが設定されているか
- TLS（HTTPS）が使用されているか
- リトライ時に指数バックオフが実装されているか

### 5. 入力検証 / インジェクション

- 外部から取得したデータ（Outlook イベントタイトル等）に制御文字のサニタイズが行われているか
- ログインジェクションの可能性がないか

### 6. トークン管理

- `AccessToken` の有効期限が適切に設定されているか（epoch 0 などの不正値でないこと）
- リフレッシュトークンの保存時にファイルパーミッションが適切か（`0600`）
- トークンが標準出力に直接出力されていないか

### 7. エラーハンドリング

- `logger.exception()` でスタックトレースが公開ログに出力されていないか（`logger.error()` で最小限のメッセージに制限すること）
- 例外メッセージに内部情報（テナント ID、トークン等）が含まれていないか

## 報告フォーマット

各問題について以下の形式で報告する:

```
### [重大度] 問題の概要

- **ファイル**: `ファイルパス:行番号`
- **問題**: 何が問題か
- **リスク**: どのようなリスクがあるか
- **推奨対策**: どう修正すべきか
```

重大度は以下の 5 段階:

| 重大度 | 基準 |
|---|---|
| Critical | 認証情報の直接的な漏洩、リモートコード実行 |
| High | サプライチェーン攻撃の経路、シークレットの間接的な露出 |
| Medium | 情報漏洩のリスク、不適切な権限設定 |
| Low | 堅牢性の問題、ベストプラクティスからの逸脱 |
| Info | 改善推奨だがリスクは最小限 |

## 出力言語

レビュー結果の最終的な出力はすべて **日本語** で記述すること。

## レビュー完了時

最後に総合評価として以下を出力する:

1. 検出された問題の重大度別件数
2. 最も優先度の高い改善項目（上位 3 件）
3. 前回レビューからの改善点（該当する場合）
